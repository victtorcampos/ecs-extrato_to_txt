## Contexto

Atualmente, os arquivos de saída (`arquivo_saida`) e os arquivos originais (`arquivo_original`) são armazenados como **Base64 em colunas Text do SQLite**. Isso causa:

- **+33% overhead** de memória (Base64 expande bytes)
- Banco de dados cresce desnecessariamente com dados binários
- Queries ficam mais lentas à medida que a tabela `lotes` acumula registros

A infraestrutura para armazenamento em disco **já existe** (`FileStoragePort` + `DiskFileStorage`), mas não está conectada ao fluxo de processamento de lotes.

### Etapa 1 — Adicionar coluna `caminho_arquivo_saida` ao modelo

**Arquivo:** `backend/src/config/models.py`

- Adicionar coluna `caminho_arquivo_saida = Column(String, nullable=True)` ao `LoteModel`
- Adicionar coluna `caminho_arquivo_original = Column(String, nullable=True)` ao `LoteModel`

**Arquivo:** `backend/src/config/database.py`

- Adicionar `ALTER TABLE` condicional em `init_db()` para criar as novas colunas (mesmo padrão já usado para `perfil_saida_id`):

  ```python
  await conn.execute(text(
      "ALTER TABLE lotes ADD COLUMN caminho_arquivo_saida TEXT"
  ))
  await conn.execute(text(
      "ALTER TABLE lotes ADD COLUMN caminho_arquivo_original TEXT"
  ))
  ```

**Arquivo:** `backend/src/application/usecases/usecases.py`

- Adicionar `file_storage: FileStoragePort` como dependência do construtor
- Na geração de saída (linha ~155-159), substituir:
  ```python
  # ANTES:
  lote.arquivo_saida = base64.b64encode(arquivo_saida.encode('utf-8')).decode('utf-8')

  # DEPOIS:
  caminho = await self.file_storage.salvar(
      conteudo=arquivo_saida.encode('utf-8'),
      nome_arquivo=f"{lote.protocolo}.txt",
      subdiretorio="saidas"
  )
  lote.caminho_arquivo_saida = caminho
  lote.arquivo_saida = None  # Não armazenar mais em Base64
  ```

**Arquivo:** `backend/src/adapters/inbound/rest/controllers/lote_controller.py`

- No endpoint de download (linhas ~252-275), implementar lógica dual:

  ```python
  if lote.caminho_arquivo_saida:
      # Novo: ler do disco
  ```
