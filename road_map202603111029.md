# Roadmap de Implementacao - Reutilizacao de Layouts por CNPJ + Mapeamento de Contas

> Ref: `proposta_202603111019.md`
> Data: 2026-03-11 10:29

---

## Contexto

O wizard de importacao (6 steps) nao oferece reutilizacao de layouts salvos para o CNPJ informado.
O usuario precisa remapear colunas e recriar regras de contas manualmente a cada importacao.
Alem disso, a pagina `LayoutDetail` nao permite editar `regras_conta` (mapeamento de contas).

**Escopo: Frontend apenas — backend ja suporta tudo (sem alteracoes necessarias).**

```
/backend   -> FastAPI + SQLAlchemy (Clean Architecture) — SEM MUDANCAS
/frontend  -> React 18 + Tailwind (FSD) — 3 arquivos a editar
```

## Consideracoes Finais

1. **Sem quebra de comportamento existente** — quando `layoutsDisponiveis` e vazio, o painel nao e renderizado. Quando `layoutAplicadoId` e `null`, `handleProcess` mantém o fluxo atual de criar novo layout.

2. **Merge de colunas por `coluna_excel`** — correlacao natural entre colunas detectadas e colunas do layout salvo. Funciona bem quando o arquivo tem a mesma estrutura (mesmo banco/ERP). Se o arquivo tiver estrutura diferente, o usuario simplesmente nao aplica o layout ou corrige manualmente na tabela.

3. **`regrasContas` pre-carregadas sao editaveis** — o `StepAccounts` nao muda. O usuario pode ajustar as regras importadas antes de prosseguir.

4. **Save separado em `LayoutDetail`** — o botao "Salvar mapeamento" persiste apenas `regras_conta`, sem sobrescrever colunas ou outras configuracoes do layout. O `...layout` spread garante que todos os campos existentes sejam mantidos.

5. **Verificar props do `AccountRulesBuilder`** — antes de implementar FASE 3, confirmar a assinatura exata do componente (`AccountRulesBuilder.jsx`) para usar os nomes corretos de props.
