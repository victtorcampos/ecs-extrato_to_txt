## Contexto

O sistema atual possui um wizard de importacao em 4 etapas (Upload > Revisao > Contas > Preview). A proposta evolui para 6 etapas, adicionando resolucao de pendencias inline e selecao de layout de saida ao fluxo, alem de regras mais expressivas (AND/OR, multiplas acoes).

**Monorepo:**
```
/backend   -> FastAPI + SQLAlchemy (Clean Architecture)
/frontend  -> React 18 + Tailwind (FSD)
```

## Consideracoes Finais

1. **Retrocompatibilidade e prioridade** - Regras existentes (sem `operador_logico`, sem `acoes`) DEVEM continuar funcionando. O `from_dict()` ja usa defaults, entao dados antigos serao carregados sem erro.

2. **Performance** - O step 4 exige um test-parse intermediario. Para arquivos grandes (>5000 linhas), considerar cache do resultado de parsing entre step 4 e step 5, evitando parse duplo.

3. **UX progressiva** - O step 3 (Contas) esta ficando complexo. Usar progressive disclosure: mostrar AND/OR e acoes multiplas somente quando o usuario clicar em "Modo avancado". A interface simples (sem AND/OR, com conta_debito/credito fixos) deve continuar como default.

4. **Migracoes de banco** - A unica mudanca de schema e `layout_id` em `MapeamentoContaModel`. Como e nullable, nao requer migracao destrutiva. SQLAlchemy com SQLite permite `ALTER TABLE ADD COLUMN` diretamente.

5. **Rotas legadas** - Manter redirect de `/perfis-saida` -> `/layouts-saida` por pelo menos uma versao para nao quebrar bookmarks ou links externos.
